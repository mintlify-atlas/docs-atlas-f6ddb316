title: "Discoverable credentials and credential management"
description: "Pico FIDO stores up to 256 discoverable credentials on-device and exposes the full CTAP 2.1 credential management API for enumeration and deletion."
---

Discoverable credentials—also called resident keys—are credentials whose private key and user metadata are stored entirely on the authenticator. A server that supports passkeys can let you sign in without ever supplying a username: the device holds everything it needs. Pico FIDO stores up to 256 resident credentials in its internal flash, exposes them through the `authenticatorCredentialManagement` command, and protects each one with the device key so they cannot be extracted.

## Discoverable credentials

When you create a credential with the `rk` option set to `true`, Pico FIDO stores a full copy of the credential on device. During authentication you can omit the `allowCredentials` list entirely; the device selects the matching credential automatically based on the relying party ID.

| Limit | Value |
|---|---|
| Maximum stored credentials | `MAX_RESIDENT_CREDENTIALS` = 256 |
| Maximum credentials returned per RP in a list | `MAX_CREDENTIAL_COUNT_IN_LIST` = 16 |
| Maximum credential ID length | `MAX_CRED_ID_LENGTH` = 1024 bytes |

## Credential management command

The `CTAP_CREDENTIAL_MGMT` command (`0x0A`) lets a PIN-authenticated client enumerate and delete resident credentials without connecting to any relying party. You must present a valid PIN UV auth token with the `CM` (`0x04`) permission to use this command.

Operations available through credential management:

- **Enumerate relying parties** — list every RP that has at least one stored credential
- **Enumerate credentials by RP** — list all credentials belonging to a specific RP, including user handle, display name, and public key
- **Delete a credential** — permanently remove a single credential by its credential ID
- **Update user information** — update the `userName` or `userDisplayName` stored with a credential

<Note>
  Deleting a credential is irreversible. The corresponding private key is erased from flash immediately.
</Note>

## credBlob — per-credential data storage

The `credBlob` extension lets you attach up to **`MAX_CREDBLOB_LENGTH` = 128 bytes** of arbitrary data to a single credential at creation time. The blob is returned in the authenticator data every time you assert that credential. Common uses include storing a username hint, a user preference blob, or a small encrypted token alongside the credential.

To use `credBlob`, pass the extension at `makeCredential` time:

```python
device.doMC(extensions={"credBlob": b"my-extra-data"}, rk=True)
```

The data is returned transparently in the assertion's authenticator data extensions map under the `credBlob` key.

## Credential counter and replay protection

Every credential assertion increments a monotonic **sign counter** (`get_sign_counter`). Relying parties can compare the counter value from successive assertions to detect cloning or replay attempts. If you use self-attestation, the counter applies globally across all credentials. With per-credential resident keys, the counter tracks usage for each credential independently.

## Device selection

The `CTAP_SELECTION` command (`0x0B`) prompts the user to tap a specific device when multiple authenticators are connected. Pico FIDO responds by lighting the LED in "press to confirm" mode and waiting for a button press, then signals the client that this device was selected.

## authenticatorSelection in registration

When calling `navigator.credentials.create()` from a browser, you can pass `authenticatorSelection` options to filter the allowed authenticator type:

```javascript
const credential = await navigator.credentials.create({
  publicKey: {
    // ...
    authenticatorSelection: {
      residentKey: "required",       // enforce discoverable credential
      userVerification: "required",  // enforce PIN
      authenticatorAttachment: "cross-platform",
    },
  },
});
```

Pico FIDO satisfies the `cross-platform` attachment and supports both `required` and `preferred` values for `residentKey` and `userVerification`.
